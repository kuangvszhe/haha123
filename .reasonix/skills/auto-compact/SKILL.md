# 自动上下文压缩 — Auto Compact

## 来源
源自 Claude Code 的 compact 系统。监控上下文窗口使用率，在达到阈值时自动压缩。

## 用法
后台自动运行。每次 API 调用前自动检查上下文使用率，按阈值触发对应级别的压缩。

## 核心原则
- 输出始终保留至少 20K token 的空间
- 压缩尽量早（60%就开始），不要等到 90% 才动手——越早压缩代价越小
- 同一段内容只压缩一次，避免反复压缩丢失信息
- 连续 3 次压缩失败 → 停止重试（死循环保护）

## 触发阈值（激进版）

| 级别 | 阈值 | Token 余量 | 行为 |
|---|---|---|---|
| 🟢 正常 | < 50% | 余量充足 | 不操作 |
| 🟡 留意 | 50-65% | 开始吃紧 | 仅清理空/无用的 tool result |
| 🟠 告警 | 65-80% | 偏紧 | 清理旧 tool result + 合并连续系统消息 |
| 🔴 紧急 | 80-90% | 很紧 | 主动压缩：总结早期对话，删无用结果 |
| ⚫ 极限 | > 90% | 极度紧张 | 强制压缩：截断早期消息，仅保留最后 N 轮 |

**关键调整**：从原来的 60%/80%/90% 三档改为 50%/65%/80%/90% 四档，更早介入，避免被动。

## 压缩策略

### 1. 微压缩（Micro Compact）
清理不需要的 tool result 内容：
```
保留最近的 N 条 tool result，删除更早的
保留 test/error 类结果（有信息价值）
删除 file_read 的完整内容（已读过的不再需要）
删除成功的 bash 命令输出（只看最后状态码）
保留失败的 bash 输出（有诊断价值）
```

### 2. 会话摘要（Summary Compact）
将早期对话压缩为摘要：
```
找到最早的关键决策点（架构选择、API设计等）
保留这些决策
删除中间的试错过程
合并为一段摘要："已确定使用 X 方案，因为..."
```

### 3. 消息截断（Truncation）
仅保留最近的核心交互：
```
保留最后 5 轮用户-助手交互
保留所有未完成的工具调用
删除最早的 N 轮交互
```

## 触发时机
- 每次 API 调用前自动检查上下文使用率
- 如果距离上次 AI 回复超过 1 小时（缓存已过期），触发时间基压缩
- 如果一条工具结果超过 5000 token，视为"大结果"需要特别处理

## 约束
- 给输出保留至少 20K token 空间
- 同一次会话已压缩过的部分不再压缩
- 压缩失败连续 3 次 → 停止重试（死循环保护）
