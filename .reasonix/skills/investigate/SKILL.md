# 根因调查

## 来源
源自 Garry Tan (gstack) 的 `/investigate` 和 Oh My Pi 的调试方法论。

## 用法
当遇到生产问题、复杂 bug、或系统故障时，系统化地追溯根因。

## 五步调查法

### 1. 收集证据
- 收集所有可用日志（应用日志、系统日志、访问日志）
- 记录问题首次发现时间和频率
- 收集复现步骤和环境信息
- 截图/录屏/错误堆栈

### 2. 时间线构建
- 按时间线排列所有事件
- 标出"第一次发现"和"最后一次无问题"的时间点
- 关联同时发生的变更（部署、配置变更、流量变化）

### 3. 假设生成
提出 3-5 个可能的根因假设，按可能性排序：
```
假设 1: [最可能] — 因为...
假设 2: [次可能] — 因为...
假设 3: [可能] — 因为...
```

### 4. 验证假设
对每个假设：
- 设计最小验证实验
- 执行验证（代码审查、日志分析、复现测试）
- 标记为「已证实」或「已排除」

### 5. 修复与验证
- 应用最小修复
- 验证修复有效
- 添加监控/告警防止复发
- 记录事后分析

## 事后分析模板
```
## 事后分析

### 概述
- 标题：
- 严重程度：
- 持续时间：
- 影响范围：

### 时间线
- [时间] 事件
- [时间] 发现
- [时间] 响应
- [时间] 修复
- [时间] 恢复

### 根因
- 直接原因：
- 根本原因：
- 为什么没被提前发现：

### 改进措施
- [ ] 监控/告警
- [ ] 代码层面
- [ ] 流程层面
- [ ] 文档层面
```
