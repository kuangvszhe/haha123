# 自进化 — Self Evolution

## 来源
源自 Hermes Agent 的 self_evolve 系统。根据使用经验自动优化自身行为和技能。

## 核心理念
每次交互都是一次学习机会。系统化地回顾、总结、优化。

## 进化周期

### 1. 采集信号
每次完成任务后收集：
- 这次踩了什么坑？（重复失败？工具用错？）
- 用户纠正了什么？（说明原有做法不对）
- 哪些步骤可以自动化？（手动操作 → 自动规则）
- 哪些 Skill 没用上？（说明它们不可达/不相关）

### 2. 生成优化建议
根据信号生成具体建议：
```
类别         | 建议示例
─────────────┼──────────────────────────
Skill 改进   | "office-hours 应该追问完自动触发 arch-review"
规则新增     | "遇到 .env 文件修改时自动 checkpoint"
工具替换     | "grep 效率低 → 改用 explore 做跨文件分析"
删除冗余     | "dap-debugger 和 debug-helper 边界不清"
```

### 3. 应用优化
- **自动修改 AGENTS.md**（更新自动触发规则）
- **创建/修改 Skill**（新增能力或改进现有）
- **更新记忆**（保存新的经验）
- 涉及破坏性改动的，先 `ask` 用户确认

## 触发时机
| 时机 | 行为 |
|---|---|
| 每次大型任务完成 | 快速回顾，产生1-2条优化建议 |
| 连续3次用户纠正 | 审查对应 Skill 并改进 |
| 每周自动审查 | 全面审查所有 Skill 的使用频率和效果 |
| 新工具/MCP 安装 | 自动创建对应 Skill 和自动触发规则 |

## 进化模式

全自动模式（已授权）。每次大型任务完成后自动执行进化流程。

### 自动执行规则
1. 任务结束后自动回顾，发现可优化点
2. 不涉及破坏性变动的 → 直接执行，末尾注明 `🤖 已自动进化：[改动摘要]`
3. 涉及破坏性变动的（删除 Skill/修改触发条件/安装 MCP）→ 仍然 ask

### 始终 ask 的边界
- 删除 Skill 或 AGENTS.md 整段内容
- 修改自动触发规则中的触发条件
- 添加新的 MCP 服务器
- 任何可能破坏现有功能的结构性变动
