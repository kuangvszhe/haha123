# 代码审查增强

## 来源
源自 Oh My Pi 的顾问模型 / 审查模型、OpenAI CodeX 的 review 能力和 Claude Code 的权限分级。

## 用法
当需要审查代码变更时，使用本 skill 指导审查流程。

## 审查清单

### 1. 正确性
- [ ] 逻辑是否完整覆盖所有分支？
- [ ] 边界条件是否处理（空值、0、负数、最大值）？
- [ ] 错误路径是否有合理处理（非吞 error）？
- [ ] 并发访问是否有竞态风险？

### 2. 安全性
- [ ] 用户输入是否经过验证/清理？
- [ ] 是否存在注入风险（SQL、命令、路径）？
- [ ] 敏感信息是否被泄露（日志、错误消息）？
- [ ] 权限检查是否到位？

### 3. 可维护性
- [ ] 命名是否清晰表达意图？
- [ ] 是否有不必要的重复代码？
- [ ] 函数/方法是否过长（>50行应考虑拆分）？
- [ ] 是否有遗留的 TODO / FIXME / 调试代码？

### 4. 性能
- [ ] 是否存在 N+1 查询问题？
- [ ] 是否有不必要的内存分配？
- [ ] 热点路径是否有性能隐患？

### 5. 测试
- [ ] 新功能是否有测试覆盖？
- [ ] 测试是否覆盖了边界条件和错误路径？
- [ ] 测试是否可独立运行（无外部依赖）？

## 审查流程

```
1. 收集变更：git diff HEAD（或等效方式）
2. 运行 review 工具做自动化审查
3. 涉及安全时运行 security_review
4. 逐文件人工过审（用 LSP 检查类型/引用）
5. 汇总发现，按严重程度排序：
   🔴 阻塞：必须修复
   🟡 建议：值得改进
   🔵 风格：仅供参考
6. 应用修复或记录决策
```

## 约束
- 审查意见必须基于具体代码，不笼统
- 不在审查中引入新功能
- 安全类阻塞项必须修复才能合入
