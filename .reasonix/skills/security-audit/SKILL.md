# 安全审计

## 来源
源自 Garry Tan (gstack) 的 `/cso`（首席安全官）、OpenClaw 的安全沙箱、Claude Code 的权限分级。

## 用法
当需要对项目或代码变更进行安全审查时使用。

## 审计流程

### 1. 资产盘点
- [ ] 哪些数据存储在系统中？
- [ ] 谁有访问权限？
- [ ] 认证机制是什么？
- [ ] 外部依赖清单

### 2. OWASP Top 10 检查
- [ ] **注入** — SQL、命令、LDAP 注入
- [ ] **失效的身份认证** — 弱密码、会话管理
- [ ] **敏感数据泄露** — 明文存储、HTTPS
- [ ] **XML 外部实体** — XXE 攻击
- [ ] **失效的访问控制** — 越权
- [ ] **安全配置错误** — 默认密码、调试模式
- [ ] **跨站脚本 XSS** — 未过滤输出
- [ ] **不安全的反序列化** — 不可信数据反序列化
- [ ] **使用已知漏洞组件** — 过时依赖
- [ ] **日志和监控不足** — 无审计日志

### 3. STRIDE 威胁建模
| 威胁 | 检查项 |
|---|---|
| **S**poofing（伪造） | 身份验证是否可绕过？ |
| **T**ampering（篡改） | 数据完整性如何保护？ |
| **R**epudiation（抵赖） | 操作是否有日志？ |
| **I**nformation Disclosure（泄露） | 敏感数据是否加密？ |
| **D**enial of Service（拒绝服务） | 有速率限制吗？ |
| **E**levation of Privilege（提权） | 权限边界是否清晰？ |

### 4. 秘密扫描
- [ ] 硬编码密钥/密码
- [ ] .env 文件是否被 git 跟踪
- [ ] 证书/私钥是否泄露
- [ ] API token 是否硬编码

### 5. 依赖审计
- [ ] 运行 `npm audit` / `pip audit` / `cargo audit`
- [ ] 检查已知漏洞（CVE）
- [ ] 最小权限原则：每个依赖真的需要吗？

## 输出格式
```
## 安全审计报告

### 严重问题 (P0) — 必须修复
- [描述] [位置] [修复建议]

### 中等问题 (P1) — 建议修复
- [描述] [位置] [修复建议]

### 低风险 (P2) — 记录在案
- [描述] [位置]

### 安全评分
- [通过/有条件通过/不通过]
```
