# 设计方案：选择性上下文注入

## 痛点

当前每次 API 调用都会把整个对话历史发给模型，包括：
- 已经解决的老问题
- 不再相关的文件内容
- 已经确认无误的代码片段
- 中间的试错过程

这导致：上下文窗口快速填满、token 浪费、价格上升。

## 核心理念

**不是所有历史都值得保留。每次发送时，只选择当前轮次需要的上下文。**

## 架构设计

```
传统方式（当前）:
  轮1: [用户问题1 + 助手回答1 + 工具结果1]
  轮2: [用户问题1 + 助手回答1 + 工具结果1 + 用户问题2 + 助手回答2]
  轮3: [用户问题1 + 助手回答1 + 工具结果1 + 用户问题2 + 助手回答2 + ...]
  → 线性增长，永远全量

选择性注入（目标）:
  轮3 向模型发送:
    ├── 当前用户问题3
    ├── 与问题3相关的历史片段（如: 之前讨论过的 API 设计决策）
    ├── 最后 N 轮交互（保持对话流畅性）
    └── 关键上下文（当前文件内容、项目结构）
  → 大小基本恒定，不随轮次增长
```

## 层级设计

```
┌────────────────────────────────────────────────┐
│  层0: 系统提示词（固定，缓存友好）               │ ← 每次都发
├────────────────────────────────────────────────┤
│  层1: 当前请求                                  │ ← 每次都发
│  - 当前用户输入                                 │
│  - 最后 2-3 轮交互                              │
├────────────────────────────────────────────────┤
│  层2: 按需注入（选择性）                         │ ← 智能选择
│  - 语义搜索相关历史片段                          │
│  - 当前操作文件的完整内容                        │
│  - 当前任务的上下文                              │
├────────────────────────────────────────────────┤
│  层3: 摘要（压缩后的历史）                       │ ← 必要时发
│  - 已解决的关键决策摘要                          │
│  - 已确认的架构设计                              │
└────────────────────────────────────────────────┘
```

## 注入决策

每次调用前，判断是否需要注入某段上下文：

```
判断维度:
  - 相关性：这段话和当前问题有关吗？（语义匹配）
  - 时效性：这段话是最近 N 轮内的吗？
  - 重要性：这段话包含关键决策/API设计吗？
  - 体积：这段话占多少 token？

决策逻辑:
  IF 相关性高 AND 时效性高 → 保留（最近 N 轮）
  IF 相关性高 AND 时效性低 → 注入摘要
  IF 相关性低 AND 时效性高 → 保留但标记可压缩
  IF 相关性低 AND 时效性低 → 丢弃或归档
```

## 需要的 Reasonix 引擎能力

| 能力 | 当前状态 | 需要做什么 |
|---|---|---|
| 逐轮标记重要/不重要 | ❌ 无 | 每轮交互后分析并标记 |
| 选择性发送上下文 | ❌ 无 | 修改 API 调用前的 prompt 组装逻辑 |
| 语义搜索历史 | ✅ 已有 MCP Memory | 知识图谱已可用 |
| 上下文摘要生成 | ⚠️ auto-compact skill | 需要引擎级集成 |

## 实施优先级

1. **P0**: 修改 prompt 组装逻辑，支持选择性发送（引擎核心改动）
2. **P1**: 每轮自动标记重要/不重要（基于 self-evolve 的经验积累）
3. **P2**: 用知识图谱 MCP 做语义搜索替代线性扫描

## 预期效果

```
100轮对话后:
当前: 100轮 × 平均500token = 50,000 token → 每次全量
目标: 层0(固定) + 层1(3轮) + 层2(按需) ≈ 5,000-8,000 token → 稳定
节省: 84%-90% 的 token 消耗
```
